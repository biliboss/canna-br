## Custo de IA — Cost Ledger

A seção acima já exibe o ledger completo e os gráficos de custo por lane. O custo total de **US$ 2,29** é dominado por **cache writes** — ao inicializar 3 lanes em paralelo com contexto completo do projeto, cada lane paga o custo de primeiro priming do cache. Execuções subsequentes da mesma missão (mesma base de código, mesmo contexto) custariam significativamente menos.

**Cache economics desta missão:**

<ul>
  <li><strong>widget-html</strong> foi a lane mais cara em tokens (1.112.243) mas também a mais complexa (remoção de campos no DOM + DOM diffing). Custo proporcional.</li>
  <li><strong>verify</strong> foi a lane mais cara em USD ($0.73) apesar de ter menos tokens — sugere que o verify fez mais leituras de contexto frio (build completo com 288 páginas).</li>
  <li>As duas lanes de domínio (<code>page-mdx</code> e <code>invite-comp</code>) foram as mais baratas e mais rápidas — escopo cirúrgico.</li>
</ul>

## Como foi coordenado

A missão usou o **Claude Code Workflow tool** com 3 lanes de trabalho paralelas seguidas de uma lane de verificação sequencial:

<ol>
  <li><strong>Decomposição</strong> — a missão foi dividida em 3 lanes de arquivo isolado (sem race condition de git): <code>page-mdx</code>, <code>invite-comp</code>, <code>widget-html</code>.</li>
  <li><strong>Execução paralela</strong> — as 3 lanes work rodaram simultaneamente, cada uma com timebox implícito de &lt;5 min. O speedup medido foi <strong>1.53×</strong> vs execução sequencial.</li>
  <li><strong>Barrier semântico</strong> — o Workflow tool aguardou todas as 3 lanes work concluírem antes de iniciar a lane verify.</li>
  <li><strong>Verify sequencial</strong> — <code>verify</code> rodou build completo (288 páginas), grep de sanidade e render check. Resultado: build green ✓.</li>
</ol>

**Premissas de coordenação aplicadas:**

<ul>
  <li>Coord roda em Opus — sub-agents (lanes) rodam em Sonnet (custo proporcional ao escopo).</li>
  <li>Cada lane é stateless e file-isolated — sem conflito de merge entre lanes paralelas.</li>
  <li>Lead time é medido no manager SurrealDB: <code>MIN(created_at)</code> → <code>MAX(closed_at)</code> da missão.</li>
  <li>Custo por lane registrado no ledger junto com token breakdown (input / output / cache_write / cache_read).</li>
</ul>
